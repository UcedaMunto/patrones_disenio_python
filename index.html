Singleton - Instancia única

El patrón Singleton en programación orientada a objetos es un patrón de diseño que garantiza que una clase solo tenga una única instancia y proporciona un punto de acceso global a esa instancia. Esto es útil cuando se necesita un control único sobre ciertos recursos, como el acceso a una base de datos, configuración de la aplicación o una conexión de red.

Características del patrón Singleton:

  • Una única instancia: La clase controla la creación de su única instancia mediante un método que verifica si ya existe una instancia. Si no existe, la crea.
  • Acceso controlado: Proporciona un punto de acceso global a esa instancia, generalmente a través de un método público o una propiedad.
  • Evita la creación directa: El constructor de la clase es privado o protegido, lo que impide que se creen instancias adicionales fuera del control de la clase.

Desventajas del patrón Singleton:

  • Dificultad en la prueba: La naturaleza global del Singleton puede complicar las pruebas unitarias, ya que puede ser difícil restablecer el estado entre las pruebas, lo que puede llevar a resultados inconsistentes.
  • Acoplamiento fuerte: Los objetos que dependen de un Singleton están acoplados a él, lo que puede dificultar la reutilización de código y la modificación de clases sin afectar a otras partes del sistema.
  • Problemas de concurrencia: En entornos multihilo, la implementación del Singleton puede requerir medidas adicionales para garantizar la seguridad de los hilos, lo que puede añadir complejidad y afectar el rendimiento.
  • Inflexibilidad: El patrón puede llevar a un diseño rígido que dificulte la extensión o modificación del comportamiento de la clase, ya que se espera que siempre haya una única instancia.
  • Uso inadecuado de recursos: Si un Singleton mantiene una gran cantidad de estado o recursos, puede causar un uso ineficiente de la memoria si la instancia no se necesita durante toda la vida de la aplicación.

Ejemplos de uso del patrón Singleton:

  • Manejo de conexión a base de datos: Un Singleton puede ser utilizado para manejar una única conexión a la base de datos, asegurando que todas las operaciones se realicen a través de esta conexión y evitando la sobrecarga de múltiples conexiones.
  • Configuración de la aplicación: Un Singleton puede contener la configuración de la aplicación, permitiendo que diferentes partes del sistema accedan a los mismos parámetros de configuración sin necesidad de crear múltiples instancias.
  • Registro (Logger): Un Singleton puede implementarse para gestionar un sistema de registro, garantizando que todos los mensajes de log se envíen a la misma instancia, lo que facilita el seguimiento y la gestión de logs a lo largo de la aplicación.

Builder - Patrón de Construcción

El patrón Builder es un patrón de diseño creacional que permite construir objetos complejos paso a paso. Separa la construcción de un objeto de su representación, lo que permite crear diferentes representaciones del mismo tipo de objeto.

Características:

  • Construcción paso a paso: El patrón permite construir un objeto de manera incremental, facilitando la creación de diferentes configuraciones.
  • Separación de la representación: La lógica de construcción se separa de la representación final del objeto, lo que permite crear diferentes variaciones.
  • Flexibilidad: Se pueden usar diferentes implementaciones de un mismo builder para crear distintas configuraciones del objeto sin modificar su código.

Desventajas:

  • Complejidad adicional: La implementación del patrón Builder puede añadir complejidad al código, especialmente si el objeto a construir es simple.
  • Aumento en el número de clases: Puede requerir la creación de múltiples clases para los builders, lo que puede resultar en una mayor cantidad de código.
  • Dependencia de la implementación: Los cambios en el objeto a construir pueden requerir modificaciones en todos los builders asociados.

Lista de clases

  • Una clase modelo - contrato para cada tipo de objeto (Los objetos comparten muchas propiedades en comun)
  • N clases builder que heredan he implementan los metodos del contrato utilizados
  • Un método o clase que utiliza los builders

Ejemplos:

  • Construcción de objetos complejos: Se utiliza para construir objetos como configuraciones de ventanas, documentos HTML, o cualquier objeto que requiera varios pasos para su creación.
  • Crear productos de diferentes configuraciones: Ideal para crear productos en un sistema de pedidos, donde los clientes pueden personalizar su pedido.
  • Formularios de entrada: Para construir formularios donde se pueden agregar campos opcionales de manera flexible.

Abstract Factory - Fábrica Abstracta

El patrón Abstract Factory es un patrón de diseño creacional que se enfoca en la creación de familias de objetos relacionados. Imagina una fábrica que produce diferentes tipos de productos, como sillas, mesas y sofás, pero en diversos estilos (moderno, clásico, rústico). Este patrón te permite crear estas familias de productos sin tener que preocuparte por los detalles específicos de cómo se construyen cada uno de ellos.

¿Por qué usar Abstract Factory?
  • Abstracción: Separa la creación de objetos de su uso, lo que hace que el código sea más flexible y fácil de mantener.
  • Consistencia: Garantiza que los objetos creados pertenezcan a la misma familia, es decir, tienen un estilo o tema en común.
  • Extensibilidad: Permite agregar nuevas familias de productos sin modificar el código existente.
  • Configuración: Facilita la configuración de un sistema en tiempo de ejecución al permitir cambiar la fábrica utilizada.
Estructura del Patrón

El patrón Abstract Factory típicamente consta de los siguientes elementos:

  • AbstractFactory: Una interfaz que declara los métodos para crear los diferentes tipos de productos.
  • ConcreteFactory: Clases concretas que implementan la interfaz AbstractFactory y crean familias de productos específicos.
  • AbstractProduct: Una interfaz que declara la operación común de todos los productos de una familia.
  • ConcreteProduct: Clases concretas que implementan la interfaz AbstractProduct y representan los productos reales.
Características:
  • Familias de productos: Facilita la creación de familias de productos que están relacionados o dependen entre sí.
  • Desacoplamiento: Separa la lógica de creación de los productos del código cliente, promoviendo el principio de inversión de dependencias.
  • Flexibilidad: Permite que el cliente trabaje con cualquier fábrica concreta sin conocer su implementación.

Desventajas:

  • Complejidad: El uso del patrón Abstract Factory puede aumentar la complejidad del código debido al número de clases e interfaces involucradas.
  • Dificultad en cambios: Agregar nuevos productos a la familia puede requerir modificar todas las fábricas abstractas y concretas.

Ejemplos:

  • Sistemas de interfaz gráfica (GUI): Usado para crear diferentes tipos de interfaces (Windows, macOS, Linux) sin que el código del cliente dependa de una plataforma específica.
  • Creación de objetos relacionados: Aplicable cuando se necesita asegurarse de que los productos creados pertenecen a la misma familia y pueden funcionar bien juntos.
  • Sistemas de bases de datos: Permite crear diferentes objetos de conexión a bases de datos sin depender de una implementación específica.

Bridge - Puente

El patrón Bridge es un patrón estructural que separa la abstracción de su implementación para que ambas puedan variar independientemente. Este patrón es útil cuando un sistema puede tener múltiples dimensiones de variación, permitiendo que cada dimensión cambie sin afectar a las otras.

¿Por qué usar Bridge?
  • Separación de responsabilidades: Divide una clase grande o un conjunto de clases relacionadas en dos jerarquías, abstracción e implementación, que pueden desarrollarse independientemente.
  • Extensibilidad: Permite añadir nuevas implementaciones sin modificar el código de la abstracción ni sus derivados.
  • Flexibilidad: Facilita la integración de múltiples plataformas o sistemas sin modificar el código principal.
Estructura del Patrón

El patrón Bridge consta de los siguientes elementos:

  • Abstraction: Define la interfaz de alto nivel que interactúa con el cliente. Usa un objeto Implementor para delegar las operaciones reales.
  • RefinedAbstraction: Extiende la clase Abstraction y agrega comportamiento adicional.
  • Implementor: Declara la interfaz de la implementación que la abstracción usará.
  • ConcreteImplementor: Proporciona la implementación concreta del Implementor.
Características:
  • Separación: Desacopla la abstracción de su implementación.
  • Desarrollo independiente: Facilita el desarrollo de la abstracción y la implementación de manera separada.
  • Reducción de complejidad: Evita que las clases crezcan demasiado y se vuelvan difíciles de gestionar.

Desventajas:

  • Complejidad adicional: Puede introducir más clases e interfaces de las necesarias si no se usa adecuadamente.
  • Dificultad en la comprensión: Puede ser más difícil de entender para los desarrolladores que no están familiarizados con el patrón.

Ejemplos:

  • Interfaces gráficas: Aplicable cuando una interfaz gráfica necesita variar en múltiples plataformas (Windows, macOS, Linux).
  • Drivers: Utilizado en la creación de drivers de hardware para gestionar diferentes implementaciones sin modificar la abstracción principal.
  • Sistemas multimedia: Implementación de distintos códecs para diferentes formatos de audio o video sin modificar la lógica de reproducción.

Composite - Composición

El patrón Composite es un patrón estructural que permite tratar a los objetos individuales y a las composiciones de objetos de manera uniforme. Este patrón se utiliza para representar jerarquías de objetos de forma que los clientes puedan interactuar con ellos sin preocuparse por las diferencias entre objetos simples y compuestos.

¿Por qué usar Composite?
  • Simplificación: Permite trabajar con estructuras jerárquicas como si fueran objetos individuales.
  • Escalabilidad: Facilita el manejo de estructuras de datos complejas al permitir la composición recursiva.
  • Consistencia: Proporciona una interfaz común para trabajar con objetos simples y compuestos.
Estructura del Patrón

El patrón Composite consta de los siguientes elementos:

  • Component: Define la interfaz común para los objetos de la composición.
  • Leaf: Representa los objetos individuales que no tienen hijos.
  • Composite: Representa los objetos compuestos que pueden contener otros objetos (tanto hojas como otros compuestos).
  • Client: Interactúa con los objetos a través de la interfaz Component.
Características:
  • Jerarquía uniforme: Simplifica las operaciones al tratar objetos simples y compuestos de la misma manera.
  • Flexibilidad: Permite construir estructuras complejas a partir de componentes más simples.
  • Reutilización: Facilita la reutilización de componentes y subestructuras dentro de la composición.

Desventajas:

  • Complejidad: Puede añadir complejidad si las operaciones sobre la jerarquía no están bien definidas.
  • Dificultad en la implementación: El manejo de las relaciones entre objetos puede ser desafiante.

Ejemplos:

  • Sistemas de archivos: Donde los archivos y carpetas se tratan de manera uniforme como nodos de una jerarquía.
  • Interfaces gráficas: Componentes de UI como paneles, botones y textos organizados jerárquicamente.
  • Organización empresarial: Representar jerarquías como departamentos, equipos y empleados.

Decorator - Decorador

El patrón Decorator es un patrón estructural que permite agregar comportamientos adicionales a objetos de forma dinámica, sin modificar su código fuente. Esto se logra mediante la envoltura de los objetos originales en decoradores.

¿Por qué usar Decorator?
  • Flexibilidad: Agrega funcionalidades a un objeto de manera dinámica sin afectar a otros objetos.
  • Reutilización de código: Permite componer comportamientos reutilizables.
  • Separación de responsabilidades: Divide las responsabilidades en diferentes decoradores en lugar de usar herencia.
Estructura del Patrón

El patrón Decorator consta de los siguientes elementos:

  • Component: Define la interfaz que implementarán tanto los objetos reales como los decoradores.
  • ConcreteComponent: Clase que define el objeto principal al que se pueden agregar funcionalidades.
  • Decorator: Clase abstracta que implementa la misma interfaz que Component y contiene una referencia al Component que envuelve.
  • ConcreteDecorator: Agrega comportamientos específicos al objeto que envuelve.
Características:
  • Composición dinámica: Permite agregar y combinar comportamientos sin modificar el código existente.
  • Desacoplamiento: Separa las funcionalidades adicionales de la lógica principal del objeto.
  • Facilidad de extensión: Facilita la adición de nuevas funcionalidades sin alterar el código base.

Desventajas:

  • Complejidad adicional: Puede incrementar el número de clases y objetos, haciendo el sistema más complejo.
  • Orden de aplicación: Los resultados pueden depender del orden en que se aplican los decoradores.

Ejemplos:

  • Interfaz de usuario: Agregar estilos o comportamientos a elementos gráficos de forma dinámica.
  • Archivos: Lectura y escritura de datos con filtros o transformaciones dinámicas.
  • Seguridad: Aplicar validaciones adicionales o verificaciones de acceso a datos.

Facade - Fachada

El patrón Facade es un patrón estructural que proporciona una interfaz simplificada para un subsistema complejo. Es especialmente útil para reducir la complejidad de las interacciones entre diferentes componentes del sistema.

¿Por qué usar Facade?
  • Simplificación: Ofrece una interfaz simple y clara para un subsistema complejo.
  • Desacoplamiento: Reduce la dependencia directa entre los clientes y los componentes del subsistema.
  • Organización: Facilita el mantenimiento del sistema al centralizar el acceso al subsistema.
Estructura del Patrón

El patrón Facade consta de los siguientes elementos:

  • Facade: Proporciona una interfaz simplificada para interactuar con el subsistema.
  • Subsystem Classes: Representan las clases reales que realizan el trabajo dentro del subsistema.
  • Cliente: Utiliza la interfaz de la fachada para interactuar con el subsistema sin conocer los detalles internos.
Características:
  • Abstracción: Oculta la complejidad del subsistema al cliente.
  • Flexibilidad: Permite modificar el subsistema sin impactar directamente a los clientes.
  • Mantenimiento: Simplifica el mantenimiento del sistema al reducir las dependencias directas.

Desventajas:

  • Dependencia en la fachada: Si la fachada falla, puede afectar el acceso al subsistema.
  • Ocultamiento excesivo: Puede dificultar la utilización de funcionalidades avanzadas del subsistema.

Ejemplos:

  • APIs: Proveer una API sencilla para operaciones complejas en un sistema.
  • Sistemas de seguridad: Centralizar el acceso a varias operaciones de autenticación y autorización.
  • Frameworks: Simplificar el uso de librerías o frameworks complejos mediante una interfaz más sencilla.
