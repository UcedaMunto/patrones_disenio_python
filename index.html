Singleton - Instancia única

El patrón Singleton en programación orientada a objetos es un patrón de diseño que garantiza que una clase solo tenga una única instancia y proporciona un punto de acceso global a esa instancia. Esto es útil cuando se necesita un control único sobre ciertos recursos, como el acceso a una base de datos, configuración de la aplicación o una conexión de red.

Características del patrón Singleton:

  • Una única instancia: La clase controla la creación de su única instancia mediante un método que verifica si ya existe una instancia. Si no existe, la crea.
  • Acceso controlado: Proporciona un punto de acceso global a esa instancia, generalmente a través de un método público o una propiedad.
  • Evita la creación directa: El constructor de la clase es privado o protegido, lo que impide que se creen instancias adicionales fuera del control de la clase.

Desventajas del patrón Singleton:

  • Dificultad en la prueba: La naturaleza global del Singleton puede complicar las pruebas unitarias, ya que puede ser difícil restablecer el estado entre las pruebas, lo que puede llevar a resultados inconsistentes.
  • Acoplamiento fuerte: Los objetos que dependen de un Singleton están acoplados a él, lo que puede dificultar la reutilización de código y la modificación de clases sin afectar a otras partes del sistema.
  • Problemas de concurrencia: En entornos multihilo, la implementación del Singleton puede requerir medidas adicionales para garantizar la seguridad de los hilos, lo que puede añadir complejidad y afectar el rendimiento.
  • Inflexibilidad: El patrón puede llevar a un diseño rígido que dificulte la extensión o modificación del comportamiento de la clase, ya que se espera que siempre haya una única instancia.
  • Uso inadecuado de recursos: Si un Singleton mantiene una gran cantidad de estado o recursos, puede causar un uso ineficiente de la memoria si la instancia no se necesita durante toda la vida de la aplicación.

Ejemplos de uso del patrón Singleton:

  • Manejo de conexión a base de datos: Un Singleton puede ser utilizado para manejar una única conexión a la base de datos, asegurando que todas las operaciones se realicen a través de esta conexión y evitando la sobrecarga de múltiples conexiones.
  • Configuración de la aplicación: Un Singleton puede contener la configuración de la aplicación, permitiendo que diferentes partes del sistema accedan a los mismos parámetros de configuración sin necesidad de crear múltiples instancias.
  • Registro (Logger): Un Singleton puede implementarse para gestionar un sistema de registro, garantizando que todos los mensajes de log se envíen a la misma instancia, lo que facilita el seguimiento y la gestión de logs a lo largo de la aplicación.

Builder - Patrón de Construcción

El patrón Builder es un patrón de diseño creacional que permite construir objetos complejos paso a paso. Separa la construcción de un objeto de su representación, lo que permite crear diferentes representaciones del mismo tipo de objeto.

Características:

  • Construcción paso a paso: El patrón permite construir un objeto de manera incremental, facilitando la creación de diferentes configuraciones.
  • Separación de la representación: La lógica de construcción se separa de la representación final del objeto, lo que permite crear diferentes variaciones.
  • Flexibilidad: Se pueden usar diferentes implementaciones de un mismo builder para crear distintas configuraciones del objeto sin modificar su código.

Desventajas:

  • Complejidad adicional: La implementación del patrón Builder puede añadir complejidad al código, especialmente si el objeto a construir es simple.
  • Aumento en el número de clases: Puede requerir la creación de múltiples clases para los builders, lo que puede resultar en una mayor cantidad de código.
  • Dependencia de la implementación: Los cambios en el objeto a construir pueden requerir modificaciones en todos los builders asociados.

Ejemplos:

  • Construcción de objetos complejos: Se utiliza para construir objetos como configuraciones de ventanas, documentos HTML, o cualquier objeto que requiera varios pasos para su creación.
  • Crear productos de diferentes configuraciones: Ideal para crear productos en un sistema de pedidos, donde los clientes pueden personalizar su pedido.
  • Formularios de entrada: Para construir formularios donde se pueden agregar campos opcionales de manera flexible.

Abstract Factory - Fábrica Abstracta

El patrón Abstract Factory es un patrón de diseño creacional que proporciona una interfaz para crear familias de objetos relacionados sin especificar sus clases concretas. Permite que un sistema sea independiente de cómo se crean, componen y representan los productos.

Características:

  • Familias de productos: Facilita la creación de familias de productos que están relacionados o dependen entre sí.
  • Desacoplamiento: Separa la lógica de creación de los productos del código cliente, promoviendo el principio de inversión de dependencias.
  • Flexibilidad: Permite que el cliente trabaje con cualquier fábrica concreta sin conocer su implementación.

Desventajas:

  • Complejidad: El uso del patrón Abstract Factory puede aumentar la complejidad del código debido al número de clases e interfaces involucradas.
  • Dificultad en cambios: Agregar nuevos productos a la familia puede requerir modificar todas las fábricas abstractas y concretas.

Ejemplos:

  • Sistemas de interfaz gráfica (GUI): Usado para crear diferentes tipos de interfaces (Windows, macOS, Linux) sin que el código del cliente dependa de una plataforma específica.
  • Creación de objetos relacionados: Aplicable cuando se necesita asegurarse de que los productos creados pertenecen a la misma familia y pueden funcionar bien juntos.
  • Sistemas de bases de datos: Permite crear diferentes objetos de conexión a bases de datos sin depender de una implementación específica.
